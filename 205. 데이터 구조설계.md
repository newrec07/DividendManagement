## 205\. 데이터 구조설계

\# 데이터 구조 설계 v1.1  
<br>
\---  
<br>
\## 전체 데이터 분류 원칙  
\`\`\`  
STATIC DATA    → 과거 재무/배당 데이터. 1회 수집 후 파일로 보관  
LIVE DATA      → 현재가, 기술지표, VIX 등. 매주 금요일 갱신 (덮어쓰기)  
PORTFOLIO DATA → 사용자 매수/매도 이력. PostgreSQL 저장 및 업데이트  
SCORE CACHE    → 매주 금요일 점수 계산 결과. PostgreSQL 저장 (공용 테이블)  
\`\`\`

\### 종목 마스터 파일 구조  
\`\`\`json  
// /data/static/tickers.json  
{  
  "tickers": \[  
    {  
      "ticker": "ABBV",  
      "name": "AbbVie Inc.",  
      "sector": "Healthcare",  
      "added\_date": "2025-01-01",  
      "currency": "USD",  
      "dividend\_frequency": "quarterly",  
      "sector\_de\_ratio": 1.8,  
      "wacc": 0.08  
    }  
  \]  
}  
\`\`\`  
<br>
※ \`sector\_de\_ratio\`: L1-F6 부채비율 계산 기준값. 업종 평균 D/E 수동 입력. 연 1회 갱신.  
※ \`wacc\`: L1-F8 ROIC vs WACC 계산 기준값. 수동 입력. 연 1회 갱신 (금리 환경 변화 시).  
※ 두 항목 모두 Bootstrap 시 초기값 입력 필수.

\## PART 2 — STATIC DATA (1회 수집, JSON 파일 보관)  
변하지 않는 과거 데이터. 최초 1회 수집 후 파일로 저장. 재수집 불필요.  
<br>
\### 2-1. 과거 배당 이력  
\`\`\`  
파일: /data/static/{TICKER}/dividends.json  
보존 기간: 과거 10년  
필드:  
  - ex\_date          배당락일  
  - pay\_date         실제 지급일  
  - dps              주당배당금  
  - is\_special       특별배당 여부 (bool)  
                     판별 기준:  
                     1) dps > 직전 4분기 평균 DPS × 3.0  
                     2) FMP label 필드에 "special" 키워드 포함  
                     3) dps > dps\_prev\_quarter × 3.0 AND dps\_prev\_quarter > 0  
                     → 3개 조건 중 하나라도 해당 시 TRUE  
  - fiscal\_quarter   해당 회계분기  
\`\`\`  
<br>
\### 2-2. 과거 재무제표  
\`\`\`  
파일: /data/static/{TICKER}/financials.json  
보존 기간: 분기별 10년 / 재무상태표 연간 10년  
수집 소스: FMP 5년 + SEC EDGAR 6~10년치 보완  
<br>
정규화 레이어 (EDGAR+FMP 접합 시 필수):  
  - 컬럼명: EDGAR XBRL 태그 → 표준 컬럼명 매핑 테이블 적용  
  - 단위: 모두 달러(USD) 기준으로 통일 (EDGAR 천 달러 단위 주의)  
  - 회계연도: 캘린더 기준으로 통일  
  - 접합 검증: 5년차 값이 FMP/EDGAR 두 소스에서 유사한지 cross-check 필수  
<br>
손익계산서 (분기별):  
  - revenue           매출  
  - eps               주당순이익  
  - operating\_income  영업이익  
  - interest\_expense  이자비용  
<br>
현금흐름표 (분기별):  
  - fcf               잉여현금흐름 (TTM 기준 연산은 엔진에서 처리)  
  - dividends\_paid    총배당지급액  
<br>
재무상태표 (연간):  
  - retained\_earnings  이익잉여금  
  - total\_debt         총부채  
  - total\_equity       자본총계  
  - shares\_outstanding 유통주식수  
\`\`\`  
<br>
\### 2-3. 과거 주가 및 기술지표  
\`\`\`  
파일: /data/static/{TICKER}/price\_history.json  
보존 기간: 과거 5년 주봉  
수집 소스: FMP 주가 원시값 → pandas-ta로 기술지표 계산  
<br>
필드:  
  - date  
  - close      주봉 종가  
  - sma\_13w    13주 이동평균  
  - sma\_40w    40주 이동평균  
  - rsi\_14w    14주 RSI  
  - bb\_upper   볼린저밴드 상단  
  - bb\_lower   볼린저밴드 하단  
  - high\_52w   52주 고점 (rolling)  
<br>
※ 점수 엔진 소스 매핑:  
   현재값(최신 주봉) → snapshot.json 에서 읽기  
   과거 시계열 필요 시 → price\_history.json 에서 읽기  
\`\`\`  
<br>
\### 2-4. 밸류에이션 밴드  
\`\`\`  
파일: /data/static/{TICKER}/valuation\_bands.json  
필드:  
  - pe\_min\_5y       5년 최저 PER  
  - pe\_max\_5y       5년 최고 PER  
  - yield\_min\_5y    5년 최저 배당수익률  
  - yield\_max\_5y    5년 최고 배당수익률  
  - roic\_10y\_avg    10년 평균 ROIC  
\`\`\`

\## PART 3 — LIVE DATA (매주 금요일 갱신, JSON 덮어쓰기)  
매주 금요일 장 마감 후 갱신. 이전 값 덮어쓰기. 히스토리 불필요.  
<br>
\### 3-1. 종목별 스냅샷  
\`\`\`  
파일: /data/live/{TICKER}/snapshot.json  
현재 시장 데이터:  
  - snapshot\_date    기준일  
  - current\_price    현재 주가  
  - current\_yield    현재 배당수익률  
  - current\_pe       현재 PER  
  - high\_52w         52주 고점 (rolling)  
<br>
최신 기술지표 (점수 엔진 현재값 소스):  
  - sma\_13w  
  - sma\_40w  
  - rsi\_14w  
  - bb\_upper  
  - bb\_lower  
  - bb\_mid  
\`\`\`  
<br>
\### 3-2. 시장 공통 데이터  
\`\`\`  
파일: /data/live/market.json  
  - snapshot\_date  
  - vix\_current  
  - vix\_updated\_at  
\`\`\`  
<br>
\### 3-3. Forward 컨센서스 (204 시뮬레이션용)  
\`\`\`  
파일: /data/live/{TICKER}/consensus.json  
  - eps\_forward\_q1~q8    향후 8분기 EPS 컨센서스  
  - eps\_revision\_4w      최근 4주 리비전 방향 (UP/NEUTRAL/DOWN)  
  - eps\_high             컨센서스 High (블렌딩 가중치 w 계산용)  
  - eps\_low              컨센서스 Low  (블렌딩 가중치 w 계산용)  
  - dps\_forward          직접 DPS 추정치 (있는 종목만)  
  - payout\_ratio\_avg5y   과거 5년 평균 배당성향

\## PART 4 — PORTFOLIO DATA (PostgreSQL — 사용자별)  
사용자가 직접 입력하고 업데이트하는 데이터. Supabase PostgreSQL로 관리.  
<br>
\`\`\`  
DB: Supabase PostgreSQL  
\`\`\`  
<br>
\### 테이블 구조  
<br>
\*\*positions (매수/매도 이력)\*\*  
\`\`\`sql  
CREATE TABLE positions (  
    id               TEXT PRIMARY KEY,  
    user\_id          UUID NOT NULL REFERENCES users(id),  
    ticker           TEXT NOT NULL,  
    type             TEXT NOT NULL,   -- BUY / SELL  
    date             TEXT NOT NULL,  
    shares           REAL NOT NULL,  
    price\_per\_share  REAL NOT NULL,  
    total\_amount     REAL NOT NULL,  
    note             TEXT  
);  
\`\`\`  
<br>
\*\*dividend\_received (배당 수령 이력)\*\*  
\`\`\`sql  
CREATE TABLE dividend\_received (  
    id             TEXT PRIMARY KEY,  
    user\_id        UUID NOT NULL REFERENCES users(id),  
    ticker         TEXT NOT NULL,  
    pay\_date       TEXT NOT NULL,  
    dps            REAL NOT NULL,  
    shares\_held    REAL NOT NULL,  
    total\_received REAL NOT NULL  
);  
\`\`\`  
<br>
\*\*watchlist (관심 종목 — 사용자별)\*\*  
\`\`\`sql  
CREATE TABLE watchlist (  
    id         TEXT PRIMARY KEY,  
    user\_id    UUID NOT NULL REFERENCES users(id),  
    ticker     TEXT NOT NULL,  
    added\_date TEXT NOT NULL  
);  
\`\`\`  
<br>
\*\*포트폴리오 집계는 쿼리로 실시간 계산\*\*  
\`\`\`sql  
\-- 종목별 평균단가 및 보유수량  
SELECT  
    ticker,  
    SUM(CASE WHEN type='BUY' THEN shares ELSE -shares END) AS total\_shares,  
    SUM(CASE WHEN type='BUY' THEN total\_amount ELSE 0 END) /  
    SUM(CASE WHEN type='BUY' THEN shares ELSE 0 END) AS avg\_cost  
FROM positions  
WHERE user\_id = :user\_id  
GROUP BY ticker;  
\`\`\`

\## PART 5 — SCORE CACHE (PostgreSQL — 공용 테이블)  
매주 금요일 점수 계산 결과 저장. 종목당 최신 1건만 유지 (upsert).  
점수는 공용 데이터 기반이므로 사용자별 중복 계산 없이 공용 테이블로 관리.  
<br>
\`\`\`sql  
CREATE TABLE score\_cache\_global (  
    ticker            TEXT PRIMARY KEY,  
    snapshot\_date     TEXT NOT NULL,  
<br>
    -- 201 정밀형  
    score\_201\_total   INTEGER,  
    score\_201\_f1      INTEGER,  
    score\_201\_f2      INTEGER,  
    score\_201\_f3      INTEGER,  
    score\_201\_f4      INTEGER,  
    score\_201\_f5      INTEGER,  
    score\_201\_f6      INTEGER,  
    score\_201\_f7      INTEGER,  
    score\_201\_f8      INTEGER,  
    score\_201\_f9      INTEGER,  
    score\_201\_f10     INTEGER,  
    signal\_201        TEXT,      -- 'STRONG\_BUY' / 'WATCH' / 'HOLD' / 'C2\_BLOCKED'  
    c2\_blocked        BOOLEAN,   -- F3=0 OR F4=0 시 TRUE  
<br>
    -- 202 심플형  
    score\_202\_total   INTEGER,  
    signal\_202        TEXT,      -- 'IMMEDIATE\_BUY' / 'WATCH' / 'WAIT'  
<br>
    -- 203 지속가능성  
    score\_203\_total   INTEGER,  
    red\_flag\_count    INTEGER,  
    yellow\_flag\_count INTEGER,  
    grade\_203         TEXT,      -- 'STRONG' / 'CAUTION' / 'DANGER'  
    red\_flag\_history  TEXT,      -- 이전 Red Flag 이력 (해제일 포함)  
<br>
    updated\_at        TEXT  
);  
\`\`\`  
<br>
※ 사용자별 watchlist는 별도 테이블로 분리 (PART 4 참조)  
※ score\_cache\_global은 전체 사용자 공용. 10명이 같은 종목을 보더라도 1번만 계산.

\## PART 6 — Bootstrap 체크포인트 관리  
\`\`\`  
파일: /data/bootstrap\_progress.json  
<br>
구조:  
{  
  "completed": \["ABBV", "HD", "MCD"\],  
  "pending": \["PLD", "WSO", "CVX", ...\],  
  "last\_run": "2025-01-10",  
  "status": "IN\_PROGRESS"  // DONE / IN\_PROGRESS / FAILED  
}  
<br>
재실행 시:  
  → completed 목록 스킵 (파일 존재 여부가 아닌 명시적 체크포인트 기준)  
  → pending 목록부터 이어서 실행  
  → 파일이 불완전하게 생성된 경우(수집 중 크래시)도 안전하게 처리  
\`\`\`

\## PART 7 — 기술 스택 확정  
\`\`\`  
과거 Static 데이터  → JSON 파일 (/data/static/)  
현재 Live 데이터    → JSON 파일 (/data/live/), 주 1회 덮어쓰기  
포트폴리오 데이터   → PostgreSQL (Supabase, 사용자별)  
점수 캐시           → PostgreSQL (Supabase, 공용 테이블)  
API 서버            → Python FastAPI (Railway)  
데이터 수집         → Python + FMP + SEC EDGAR + Alpha Vantage  
스케줄러            → Railway Cron Job (APScheduler 대체, 슬립 모드 방지)  
프론트엔드          → React + Tailwind + Recharts + TanStack Table  
\`\`\`
